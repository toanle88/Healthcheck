# Lesson 07: Entra ID Passwordless Security 🔑🛡️

In modern cloud environments, storing passwords, API keys, or connection strings in configuration files or key vaults represents a significant security risk. If a developer commits a secret by accident, or a server is compromised, the credentials are leaked.

This project implements a **100% passwordless and secretless** architecture using **Microsoft Entra ID (formerly Azure Active Directory)**.

---

## 1. Managed Identities: Computing Without Passwords

In Azure, our container applications (API, Web, and Worker) run under a **User-Assigned Managed Identity**. This identity functions as an Azure service account that is dynamically attached to the computing resource.

Instead of authenticating with a client secret (password), the Azure host environment provides a local token endpoint. Our applications use the Azure SDK's `DefaultAzureCredential` to communicate with this endpoint, retrieve a short-lived JSON Web Token (JWT), and present it to downstream services.

---

## 2. Passwordless PostgreSQL Connections (`internal/store/postgres.go`)

Our PostgreSQL Flexible Server does not verify passwords. Instead, it is integrated with Microsoft Entra ID.

In [postgres.go](file:///mnt/d/Dev/Projects/Healthcheck/internal/store/postgres.go), we configure the `BeforeConnect` hook on the PostgreSQL connection pool (`pgxpool`):

```go
cred, err := azidentity.NewDefaultAzureCredential(nil)
// ...
cfg.BeforeConnect = func(ctx context.Context, pgc *pgx.ConnConfig) error {
    token, err := cred.GetToken(ctx, policy.TokenRequestOptions{
        Scopes: []string{"https://ossrdbms-aad.database.windows.net/.default"},
    })
    if err != nil {
        return fmt.Errorf("failed to get azure ad token: %w", err)
    }
    pgc.Password = token.Token // Inject the token as the temporary password!
    return nil
}
```

### How the Hook Works
1. When the app needs to establish a new connection to Postgres, the `BeforeConnect` hook triggers.
2. `cred.GetToken` requests a fresh access token scoped specifically for Azure Database for PostgreSQL (`https://ossrdbms-aad.database.windows.net/`).
3. Under the hood, `DefaultAzureCredential` uses the local Managed Identity endpoint in production, or uses credentials from `az login` in local development.
4. The retrieved JWT is assigned to `pgc.Password`.
5. PostgreSQL validates the JWT against Entra ID, authorizes the user, and opens the connection. 
6. Because tokens expire, this hook fetches a fresh token for *each* new connection, removing the risk of expired credentials.

## 3. OIDC Federated Identity (Secretless CI/CD)

Deploying infrastructure or code from GitHub to Azure traditionally required copying an Azure Service Principal secret into GitHub Repository Secrets. 

In this repository, we use **OIDC (OpenID Connect) Workload Identity Federation**. 
- The GitHub actions workflow requests a JWT token from GitHub.
- GitHub signs a JWT asserting that the run is executing for repository `toanle88/Healthcheck` on the `main` branch.
- Azure Entra ID validates the signature and cross-checks the **Federated Credential** setup.
- If it matches, Entra ID exchanges the GitHub JWT for a short-lived (1-hour) Azure access token, allowing the deployment to run without any permanent passwords saved in GitHub.

## 4. API Authentication & RBAC Middleware (`internal/middleware/auth.go`)

Our HTTP REST endpoints are secured by custom middleware. The React frontend obtains a user token from Entra ID and sends it via the `Authorization: Bearer <token>` header.

The verification in [auth.go](file:///mnt/d/Dev/Projects/Healthcheck/internal/middleware/auth.go) performs four validation steps:

### A. JWKS Verification
Azure signs JWT tokens using asymmetric cryptography. The public keys are published in a standard **JSON Web Key Set (JWKS)** endpoint. Our middleware fetches and caches these keys to verify the signature of incoming tokens without making a network call for every single request:
```go
jwksURL := fmt.Sprintf("https://%s.ciamlogin.com/%s/discovery/v2.0/keys", tenantID, tenantID)
```

### B. Claim Validation
We parse the token claims and verify:
- **`iss` (Issuer)**: Verifies the token was generated by our configured Entra tenant.
- **`aud` (Audience)**: Verifies the token was generated specifically for our API's Client ID.
- **`exp` (Expiration)**: Enforces that the token is not expired.

### C. Role & Scope Enforcement (`RequireRoleOrScope`)
Once authenticated, we restrict API mutations (e.g., adding or deleting health check targets) to users holding admin privileges:
```go
func RequireRoleOrScope(allowedRoles []string, allowedScopes []string) gin.HandlerFunc {
    return func(c *gin.Context) {
        roles, _ := c.Get("roles")
        scopes, _ := c.Get("scopes")
        // Check if claims contain any of the allowedRoles or allowedScopes...
    }
}
```
In [main.go](file:///mnt/d/Dev/Projects/Healthcheck/cmd/api/main.go#L204), this middleware is attached to mutation routes:
```go
adminRequired := middleware.RequireRoleOrScope([]string{"Healthcheck.Admin"}, nil)
api.POST("/targets", adminRequired, h.CreateTarget)
api.DELETE("/targets/:id", adminRequired, h.DeleteTarget)
```
Only users with the `"Healthcheck.Admin"` role mapped in the Entra ID enterprise application can add or remove targets.
